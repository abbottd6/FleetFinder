What is FleetFinder?

FleetFinder is a companion app for the MMO video game Star Citizen, a space simulation game being developed by Cloud Imperium Games (CIG). Although Star Citizen is in an "alpha" phase, it has live and playable servers (many would dispute the validity of this description) where players can experiment and test the available features.

Given the game's current status as an alpha version, it lacks features that are considered standard for modern MMOs. One major gap is a group finder, a tool that helps players connect with others who share similar interests. The goal of FleetFinder is to fill this gap by helping players team up for shared gameplay experiences, whether collaborative, combative, or both.

Why FleetFinder?

I am building FleetFinder to apply the programming skills that I've developed in the WGU Computer Science program and push myself to branch off to learn new technologies not covered in my coursework. The app solves a real problem for the Star Citizen community while also giving me a platform to grow as a fullstack developer.

This project has given me hands-on experience designing a scalable web app while working with languages like Java and TypeScript, frameworks like Spring Boot and Angular, troubleshooting, testing, and much more. FleetFinder development is still in progress, with several features that I plan to implement in the v2/MVP for the app. Each of the features that I plan to implement moving forward will provide further opportunity for me to expand my skills and deepen my understanding of modern software development.

FleetFinder v1 (Current)

v1 of FleetFinder represents the core CRUD functionality of the app, including:

  • MySQL database containing group listings and cacheable static lookup data for listing attributes
  • RESTful APIs built with Spring Boot
  • Basic Angular frontend with reactive forms for listing creation and a table for viewing current listings
  • Responsive web design for different viewport sizes
  • Thorough unit and integration testing of services, mapper configurations, REST controllers, etc.

I will be adding user accounts in v2, so the group listing creation for v1 just uses a hard coded testUser. The UI is only partially functional in v1, it contains placeholders for features planned in future versions. This version lays the foundation for the more advanced functionality and serves as a snapshot of my progress thus far.

Roadmap

Features that I am in the process of learning for v2 of the app include setting up Jenkins for CI/CD, user accounts and authentication, securing API endpoints, real-time messaging, refactoring the group listings table using Angular material tables, and searching and filtering functionality for the group listings. These additions will round out the app for what I am considering the minimum viable product.

Beyond v2, assuming that I can get to this point before CIG add a group finder in game, I plan to implement admin moderation tools for group listings, reporting for listings, functionality for users to save templates for listings, and data analytics for listing attributes.

Architecture Overview

FleetFinder is designed with a modular full-stack architecture that separates concerns between the frontend, backend, and database layers. This structure enables better maintainability, scalability, and testability as the application grows.

Backend Architecture

  • Relational Data Model

    • The foundation of the backend is a MySQL 8.2 database structured around a normalized schema. Reusable values such as PvP statuses, gameplay categories, and planetary systems are separated into dedicated lookup tables, alongside tables for user-generated data like user accounts and group listings. I chose this approach primarily in order to simplify searching and filtering by requiring normalized attributes for listings, but also to minimize redundancy and simplify future expansion or modification of these fields.

  • Entity-DTO Separation

    • Spring Data JPA maps the relational database tables to the Java entity classes, while Data Transfer Objects (DTOs) are organized into two submodules: Response DTOs expose the necessary data to the frontend and Request DTOs carry minimal identifiers from the client to the backend. This separation decouples the internal model from the API interface, enables strong input validation, and simplifies future refactoring.

  • Lookup Table Structure

    • Each lookup table is mapped to its own entity and DTO class, with corresponding repository, service interface, and service implementation. While I considered abstracting these services behind a shared generic interface, I chose to define individual service pairs to keep the logic clear, maintain separation of concerns, and allow the tables to evolve independently if necessary.

  • Object Mapping and Validation

    • ModelMapper is used to streamline conversions between entities and DTOs, with the majority of the simple lookup table conversions using implicit mappings while the more complex mappings, like group listings, use explicit ModelMapper configurations. Spring Boot's built in validation annotations help enforce data integrity at each stage before persisting new listings or returning data to the client.

  • Caching

    • Lookup data is cached on both the frontend and in the backend using EhCache to reduce redundant database queries for static reference data. This lightweight caching layer ensures that frequently requested reference data can be served quickly without repeated access to the database.

  • RESTful API Design

    • The backend exposes well-structured REST APIs using Spring Boot, providing endpoints for group listings and lookup data. The API is organized using RESTful conventions and designed to be easily extendable as new features are added.

  • Backend Testing

    • The backend also includes extensive unit and integration testing to ensure reliability, maintainability, and safe future deployments. These tests include edge cases for invalid objects as well as success and failure paths. The tests were created with the goal of integrating a Jenkins CI/CD pipeline in v2 of the app, enabling automated test runs before deploying updates and new versions to the EC2 instance.

Frontend Architecture

  • Angular SPA

    • The frontend is built with Angular 22 as a single-page application, utilizing Angular routing for page navigation and maintaining application state across views.

  • Component-Based Design

    • The UI is broken into modular, reusable components, including a shared dropdowns module that integrates directly with Angular forms. This structure improves code clarity by keeping logic encapsulated and simplifies future development (e.g. filtering search results) and testing.

  • Distinct and Typed Request/View Models

    • The frontend defines separate, strongly typed models for request and response data for group listings to mirror the backend entity and DTO structure. This distinction maintains a clear boundary between form submission logic and UI rendering.

  • Reactive Forms

    • Angular Reactive Forms are used to manage group listing creation, providing real-time validation and fine-grained control over form data and structure.

  • Dropdowns with Lookup Data

    • Form dropdowns are dynamically populated by calling backend endpoints that serve the cached lookup data. The form stores the ID values while displaying the corresponding human-readable names. Some of the fields are also context-sensitive. For example, the available gameplay subcategories are filtered based on the selected gameplay category. Another example is the event schedule fields for the date picker, time zone, and time, which become enabled or disabled based on the user's "group status" selection.

  • Time Zone Normalization

    • The Create Listing form accepts separate user inputs for the date, time, and time zone of the "event schedule" field. These inputs are combined and converted into a UTC-normalized ISO string for consistent storage in the database, regardless of the user's local time zone. The event timestamps, as well as the creation and update timestamps generated by the database, are converted back to the user's browser time for display to provide a seamless and intuitive experience across different regions.

  • Frontend Caching

    • The frontend also caches static lookup data on the client side using a generic helper method in the lookup service. This prevents repeated HTTP requests from the same user during a session and complements the backend caching strategy by reducing unnecessary network traffic and improving performance.

  • Responsive Design

    • The interface is styled with modified Bootstrap and custom CSS for a clean, responsive experience across desktop and mobile devices.

Deployment and Containerization

  • Dockerized Environment

    • The entire application is containerized using Docker, with each major service running in its own container for separation of concerns and ease of deployment:

    • MySQL - relational database
    • Spring Boot Backend - REST API server
    • Angular Frontend - compiled and served with NGINX
    • NGINX Reverse Proxy - handles HTTPS and routes traffic to backend and frontend containers
  • Docker Compose

    • Docker Compose orchestrates all services for local and production environments, enabling consistent builds and simplified development workflows.

  • SSL and HTTPS

    • The app is served over HTTPS for secure communication between client and server, using NGINX configuration for SSL termination.
